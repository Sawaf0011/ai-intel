# Contributing

## Branch naming

```
feature/short-description
fix/short-description
chore/short-description
docs/short-description
```

Use hyphens, not underscores. Keep it short — `feature/reddit-scraper`, not
`feature/add_the_new_reddit_scraper_module`.

## Commit messages

Follow [Conventional Commits](https://www.conventionalcommits.org/):

```
type(scope): short description
```

Allowed types: `feat`, `fix`, `chore`, `docs`, `refactor`, `test`, `ci`.

Examples from this repo:

```
chore: initial project skeleton with uv, ruff, and directory structure
feat: FastAPI app skeleton with config and health endpoint
feat: database models, async SQLAlchemy setup, and initial Alembic migration
feat: Docker and docker-compose setup with Postgres + pgvector and migration service
docs: README quickstart, contributing guide, CI workflow scaffold, and LICENSE
```

Keep the subject line under 72 characters. Use the body for *why*, not *what*.

## Code style

Run `make format` before committing. CI runs `make lint` on every push — a lint
failure blocks merge.

```bash
make format   # black + ruff import sort
make lint     # ruff check
```

## Adding a database migration

See the [Database migrations](README.md#database-migrations) section in the README.
Always review the autogenerated file before applying — autogenerate misses PostgreSQL
extensions and some index options.

## Tests

All new code must include tests where applicable. Run `make test` before committing.
The current suite requires no live database; keep it that way until the repository
layer is introduced.

```bash
make test     # uv run pytest -v
```

## Pull request checklist

- [ ] `make test` passes locally
- [ ] `make lint` passes locally
- [ ] README updated if user-facing behaviour changed
- [ ] Migration included (and reviewed) if schema changed
- [ ] Commit messages follow Conventional Commits
